# Monorepo 项目完全手册

> 基于 pnpm workspaces 的 TypeScript monorepo，搭配 ESLint + Prettier + Husky + Commitlint + lint-staged 全链路代码规范。

## 二、每个配置文件详解

### 2.1 `pnpm-workspace.yaml` — monorepo 的注册表

```yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

| 字段 | 含义 |
|------|------|
| `apps/*` | `apps/` 下所有目录都是子包 |
| `packages/*` | `packages/` 下所有目录也都是子包 |

这是整个 monorepo 的基石。pnpm 读到这个文件后，`pnpm install` 时会扫描这些目录，给它们之间建立软链接。

---

### 2.2 `package.json`（根）— 仓库的中央大脑

```json
{
  "name": "@kfjr-monorepo-test",
  "private": true,
  "type": "module",
  "scripts": {
    "lint": "eslint ./ --ext .ts,.js --max-warnings=0",
    "format": "prettier --config .prettierrc . --write",
    "prepare": "husky",
    "cz": "cz"
  }
}
```

| 字段 | 值 | 为什么 |
|------|-----|--------|
| `private: true` | 避免被误发到 npm | monorepo 根不能发布 |
| `type: "module"` | `.js` 文件走 ESM | eslint.config.js 用了 `import` |
| `prepare: "husky"` | `pnpm install` 后自动初始化 husky | 让 git hooks 生效 |
| `cz: "cz"` | 运行 commitizen 交互式填 commit | 不记格式，用菜单选 |

所有 devDependencies 都装在根：
- **TypeScript** — 类型系统和编译器
- **ESLint + Prettier** — 代码质量和格式
- **Husky** — Git hooks 管理
- **lint-staged** — 只检查 git 暂存区的文件
- **commitlint** — 校验 commit 信息格式
- **commitizen** — 交互式生成规范的 commit 信息

---

### 2.3 `tsconfig.base.json` — TypeScript 配置基座

```json
{
  "compilerOptions": {
    "composite": true,
    "incremental": true,
    "declaration": true,
    "declarationMap": true,
    "emitDeclarationOnly": true,
    "noEmitOnError": true,
    "skipLibCheck": true,
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "resolveJsonModule": true,
    "strict": true
  }
}
```

| 配置项 | 值 | 含义 |
|--------|-----|------|
| `strict` | `true` | 开启所有严格检查（`noImplicitAny` 等一篮子） |
| `composite` | `true` | 开启 TypeScript Project References 模式 |
| `incremental` | `true` | 增量编译，只编译改过的文件 |
| `declaration` | `true` | 生成 `.d.ts` 类型声明文件 |
| `declarationMap` | `true` | 类型声明的 source map |
| `emitDeclarationOnly` | `true` | 只出 `.d.ts`，不出 `.js` |
| `noEmitOnError` | `true` | 有类型错误就不生成文件 |
| `skipLibCheck` | `true` | 跳过 node_modules 里的 `.d.ts` 检查（提速） |
| `esModuleInterop` | `true` | 允许 `import` 方式导入 CommonJS 模块 |
| `allowSyntheticDefaultImports` | `true` | 允许 `import React from 'react'` 写法 |
| `resolveJsonModule` | `true` | 可以直接 `import` JSON 文件 |

**继承关系：**
```
tsconfig.base.json          ← 基座（所有公共配置）
    ↑ extends
tsconfig.json (根)          ← 空壳，只 extends
    ↑ extends
apps/front-app/tsconfig.json    ← 每个子包只写一句 extends
packages/utils/tsconfig.json
```

⚠️ **已知问题：** `composite: true` 开了但项目引用（`references` 数组）没配，编辑器可能会报 "Unable to get relative path" 的 TS 错误。当前项目没有 build 步骤、直接用 ts-node 跑源码，所以 `composite` 这些配置暂时用不上，可以考虑去掉。

### 2.4 `eslint.config.js` — ESLint 扁平配置

```js
import js from '@eslint/js';
import tseslint from 'typescript-eslint';
import eslintConfigPrettier from 'eslint-config-prettier';
import eslintPluginPrettier from 'eslint-plugin-prettier';

export default tseslint.config(
  js.configs.recommended,           // ESLint 官方推荐规则（no-undef 等）
  ...tseslint.configs.recommended,  // TypeScript ESLint 推荐规则
  eslintConfigPrettier,             // 关掉和 Prettier 冲突的 ESLint 规则

  { ignores: ['node_modules/**', 'dist/**', ...] },  // 忽略目录

  {
    files: ['**/*.ts', '**/*.tsx'],
    languageOptions: {
      parser: tseslint.parser,
      parserOptions: { project: true, tsconfigRootDir: import.meta.dirname },
    },
    plugins: {
      '@typescript-eslint': tseslint.plugin,
      prettier: eslintPluginPrettier,   // 把 Prettier 格式化当作 ESLint 规则
    },
    rules: {
      'prettier/prettier': 'error',     // 格式不对 = ESLint 报错
    },
  },
);
```

**三个 Prettier 相关包的职责：**

| 包 | 干什么 |
|----|--------|
| `prettier` | 格式化引擎本身 |
| `eslint-config-prettier` | 关掉 ESLint 里和格式有关的规则（避免 ESLint 和 Prettier 打架） |
| `eslint-plugin-prettier` | 把 Prettier 的检查注入为 ESLint 的一条规则（`prettier/prettier`） |

### 2.5 `.prettierrc` — 格式化规则

```json
{
  "semi": true,           // 语句末尾加分号
  "singleQuote": true,    // 字符串用单引号
  "tabWidth": 2,          // 缩进 2 空格
  "printWidth": 120,      // 一行最多 120 字符
  "endOfLine": "auto"     // 换行符照文件原样（Windows CRLF / Linux LF 不互掐）
}
```

### 2.6 `.commitlintrc.json` — Commit 信息规范

```json
{
  "extends": ["@commitlint/config-conventional"],
  "rules": {
    "scope-empty": [2, "never"]
  }
}
```

采用的格式：**Conventional Commits（约定式提交）**

```
type(scope): subject

feat(auth): 添加登录功能      ← ✅ 正确
fix(ui): 修复按钮对齐         ← ✅ 正确
feat: 添加登录               ← ❌ scope 为空，被 scope-empty 规则拦截
乱写的                       ← ❌ 没 type 没 scope
```

| type 常用值 | 什么时候用 |
|-------------|-----------|
| `feat` | 新功能 |
| `fix` | 修 bug |
| `docs` | 改文档 |
| `style` | 代码格式（改空格、分号，不影响逻辑） |
| `refactor` | 重构（不新增功能也不修 bug） |
| `test` | 加测试 |
| `chore` | 杂务（改构建、升级依赖等） |

规则 `"scope-empty": [2, "never"]` 的意思是：**scope 必填**。`[2, "never"]` = 错误级别 2（报错），不允许 scope 为空。

### 2.7 `.lintstagedrc.js` — 提交前只检查暂存文件

```js
import { ESLint } from 'eslint'

const removeIgnoredFiles = async (files) => {
  const eslint = new ESLint()
  const ignoredFiles = await Promise.all(
    files.map((file) => eslint.isPathIgnored(file))
  )
  const filteredFiles = files.filter((_, i) => !ignoredFiles[i])
  return filteredFiles.join(' ')
}

export default {
  '*': async (files) => {
    const filesToLint = await removeIgnoredFiles(files)
    return [`eslint ${filesToLint} --max-warnings=0`]
  }
}
```

**执行流程：**

```
git add 暂存文件
      ↓
lint-staged 拿到暂存文件列表（如 ['src/a.ts', 'dist/b.js']）
      ↓
removeIgnoredFiles() —— 逐个问 ESLint "这个文件要忽略吗？"
      ↓
筛掉被忽略的 → 只保留需要检查的文件
      ↓
拼成命令：eslint src/a.ts --max-warnings=0
      ↓
ESLint 通过 → 提交成功 / ESLint 不过 → 提交被拦截
```

**为什么需要 `removeIgnoredFiles`？**  
lint-staged 传给函数的文件是所有暂存文件（包括 `dist/`、`node_modules/`），但它不自动读 `.eslintignore`。不筛的话会把不该检查的文件也传给 ESLint。

### 2.8 Husky hooks — Git 提交守护

```
.git/hooks/
    ├── pre-commit    → husky 注册 → .husky/pre-commit    → npx lint-staged
    └── commit-msg    → husky 注册 → .husky/commit-msg    → npx commitlint --edit $1
```

**提交流水线：**

```
git commit -m 'xxx'
      │
      ├── ① pre-commit: lint-staged 启动
      │       ↓
      │   eslint 检查暂存文件（含 prettier/prettier 规则）
      │       ↓
      │   通过 ✅ / 不通过 ❌ → 提交中断
      │
      ├── ② commit-msg: commitlint 启动
      │       ↓
      │   校验 commit 信息是否符合 conventional commits 格式
      │       ↓
      │   通过 ✅ / 不通过 ❌ → 提交中断
      │
      └── 全部通过 → git commit 完成 ✅
```

## 三、代码共享机制（核心）

### 3.1 依赖声明

`apps/front-app/package.json`：
```json
{
  "dependencies": {
    "@kfjr-monorepo-test/utils": "workspace:*"
  }
}
```

`workspace:*` 告诉 pnpm：**"别去 npm 下载，用我仓库里的本地包。"**

### 3.2 软链接解析

pnpm 在 `apps/front-app/node_modules/` 下创建软链接：

```
apps/front-app/node_modules/@kfjr-monorepo-test/utils
        ⬇ 软链接指向
packages/utils/
```

### 3.3 入口文件

`packages/utils/package.json`：
```json
{
  "main": "./src/index.ts"
}
```

`main` 字段指向 TypeScript 源码（不是编译后的 `.js`），因为当前项目用 ts-node 直接跑 TS。

### 3.4 完整链路

```
front-app/src/index.ts
  │  import { add } from '@kfjr-monorepo-test/utils'
  │
  ├── 1. Node 在 apps/front-app/node_modules/ 下找 @kfjr-monorepo-test/utils
  ├── 2. 发现是软链接 → 跳转到 packages/utils/
  ├── 3. 读 package.json 的 "main" → ./src/index.ts
  └── 4. 导入 export function add(x, y) { return x + y; }
```

## 四、常用命令速查

| 命令 | 干什么 |
|------|--------|
| `pnpm install` | 安装所有依赖 + 初始化 husky |
| `pnpm run lint` | 全仓库 ESLint 检查 |
| `pnpm run format` | 全仓库 Prettier 格式化 |
| `pnpm run format-watch` | 监听文件改动、自动格式化 |
| `pnpm run cz` | 交互式 commit（用菜单选 type 和 scope） |
| `pnpm add -D <包名> -w` | 给根装 devDependency（`-w` 不能少） |
| `npx ts-node apps/front-app/src/index.ts` | 直接运行 TypeScript 源码 |

## 五、怎么加新包

### 加一个新的共享库（如 `packages/logger`）

```
mkdir -p packages/logger/src
```

**`packages/logger/package.json`：**
```json
{
  "name": "@kfjr-monorepo-test/logger",
  "private": true,
  "version": "0.1.0",
  "main": "./src/index.ts"
}
```

**`packages/logger/tsconfig.json`：**
```json
{
  "extends": "../../tsconfig.base.json"
}
```

**`packages/logger/src/index.ts`：**
```ts
export function log(msg: string) {
  console.log(`[LOG] ${msg}`);
}
```

### 让 front-app 用它

**`apps/front-app/package.json`：**
```json
{
  "dependencies": {
    "@kfjr-monorepo-test/utils": "workspace:*",
    "@kfjr-monorepo-test/logger": "workspace:*"
  }
}
```

然后 `pnpm install`，之后就能在 `front-app` 里 `import { log } from '@kfjr-monorepo-test/logger'`。
