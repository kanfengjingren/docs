# 监控前端的报错和上报

思路是：前端报错的时候我们去监听一下，然后设置一个函数传入监听。当有报错的时候就调用函数。所以大体分为了监控部分和上报部分

## 监控三种错误：js、网络请求、资源加载

### js错误

js执行的时候发生的错误：比如这种简单的错误

```js
const a = 1;
a= 2;
```

既然思路是再浏览器收到报错警告的时候去插入函数，那就要去看看是谁在管理错误。

#### window.onerror

window.onerror是浏览器用来捕获错误的手段，我们可以改造这个函数来达到我们的目的

#### window.unhandledrejection

window.unhandledrejection是用来捕获没有处理的promise异常的，window.onerror不能捕获到promise异常

#### 代码src/errorHandle.ts

```ts

//   src/errorHandle.ts
import { sendError } from "./sender";


export const monitorJavaScriptErrors = (reportUrl: string,
    projectName: string,
    environment: string) => {

    //监控错误，这个通过winows的error来监控
    const originalFn = window.onerror;
    window.onerror = (message, source, lineno, colno, error) => {
        const errorInfo = {
            type: 'JavaScript Error',
            message,
            source,
            lineno,
            colno,
            stack: error ? error.stack : null,
            projectName,
            environment,
            timestamp: new Date().toISOString(),
        };
        //如果触发了，那你就帮我传一下呗
        sendError(reportUrl, errorInfo);
        //如果我原来就写了，那我就帮你执行一下
        if (originalFn) {
            return originalFn(message, source, lineno, colno, error)
        }

    }

    const originalOnUnhandledRejection = window.onunhandledrejection;
    window.onunhandledrejection = (event) => {
        const errorInfo = {
            type: 'Unhandled Promise Rejection',
            message: event.reason?.message || event.reason,
            stack: event.reason?.stack,
            projectName,
            environment,
            timestamp: new Date().toISOString(),
        };
        sendError(reportUrl, errorInfo);

        // 关键点：执行原有的 Promise 错误处理逻辑
        // 这样做是为了不破坏宿主环境（例如用户自己写的或其他 SDK）已有的错误处理逻辑
        if (originalOnUnhandledRejection) {
            return originalOnUnhandledRejection.call(window, event);
        }
    }

}
```

### 网络问题

网络的问题也是一样，找到对应的捕获，然后去改造就行了。不过需要注意的是自身的问题不上报，如果上报会导致递归上报等问题。这里设置了xmlhttprequest和fetch的监控

- 先拿到XMLHttpRequest.prototype.open函数，然后将open指向我们写的函数，这个函数重我们去监听error就好了。最后去执行请求 `return originalXhrOpen.apply(this, [method, url, ...args] as any);`

```ts
import { sendError } from "./sender";

export const moniterNetWorkErrors = (reportUrl: string,
    projectName: string,
    environment: string) => {

    const originalXhrOpen = XMLHttpRequest.prototype.open;
    XMLHttpRequest.prototype.open = function (
        method: string,
        url: string | URL,
        ...args: any[]
    ) {
        //如果上报自身有问题，不应该上报，不然会递归调用出问题
        const urlStr = typeof url === 'string' ? url : String(url)
        if (urlStr.includes(reportUrl)) {
            return originalXhrOpen.apply(this, [method, url, ...args] as any);
        }

        //监听error事件，有就上报
        this.addEventListener('error', () => {
            sendError(reportUrl, {
                
                type: "Network Error",
                message: `这个${url}，发送${method}请求时失败`,
                projectName,
                environment,
                timestamp: new Date().toISOString(),
            })
        });
        return originalXhrOpen.apply(this, [method, url, ...args] as any)
    }

    //劫持fetch
    const originalFetch = window.fetch;

    window.fetch = async (input, init) => {
        // 关键点：排除上报接口自身的请求，防止死循环
        const urlStr = (input instanceof Request) ? input.url : String(input);
        if (urlStr.includes(reportUrl)) {
            return originalFetch(input, init);
        }

        try {
            const response = await originalFetch(input, init);
            if (!response.ok) {
                sendError(
                    reportUrl,
                    {
                        type: 'Fetch Error',
                        message: `HTTP ${response.status}: ${response.statusText}`,
                        url: input instanceof Request ? input.url : input,
                        projectName,
                        environment,
                        timestamp: new Date().toISOString(),
                    }
                );
            }
            return response;
        } catch (error) {
            // 网络故障等无法发出请求的情况
            sendError(
                reportUrl,
                {
                    type: 'Fetch Error',
                    message: `Fetch Failed: ${input}`,
                    projectName,
                    environment,
                }
            );
            throw error;
        }
    };

}
```

### 资源问题

img、script这些资源加载问题，window是捕捉不到的，所以需要我们额外的处理

- 原因： 因为资源加载失败产生的 error 事件是 不冒泡 的。 window.onerror 机制依赖于事件冒泡到顶层窗口，因此它对资源加载错误无能为力。

- 解决方案： 我们必须利用 addEventListener 的 捕获阶段 （将第三个参数设为 true ）。 虽然错误事件不冒泡，但在 捕获阶段 （事件从 window 向下传播到目标元素的过程），我们依然有机会在 window 层级拦截到这些错误。



```ts
import { sendError } from "./sender"


export const monitorResourceErrors  = (
    reportUrl: string,
    projectName: string,
    environment: string
)=>{
    console.log('触发了');
    
    window.addEventListener('error',(event)=>{
        const target = event.target as HTMLElement
        console.log('资源错误事件触发, target:', target, 'tagName:', target?.tagName);

        if(target && (target.tagName === 'IMG' || target.tagName === 'SCRIPT')){

            sendError(reportUrl,{
                type: "Source Error资源错误",
                message: `${target.tagName}加载失败，${target.getAttribute('src') || target.getAttribute('href')}`,
                projectName,
                environment,
                timestamp: new Date().toISOString(),
            })
        }
    },true)
}
```

这样写有个小问题，最后如果直接用src-module去引入，因为浏览器的加载时机的问题(script标签 --- 链接)，浏览器会在html全部解析完成之后才去执行script里面的代码。因此静态的img之类的资源加载问题不能被监控到

## send上报

接下来就是上报的实现，看起来也就是一个简单的发送请求。一开始使用的是fetch，后面了解了说

- 异步请求可能会被取消：页面关闭时，浏览器通常会 cancel 掉所有未完成的请求。
- 同步请求会阻塞跳转：虽然能强行发出去，但会卡住页面切换，严重影响体验。

因此出现了**Navigator.sendBeacon**这个东西

```ts
//上报错误

export const sendError = (url: string, errorInfo: any) => {

    if (navigator.sendBeacon) {
        const blob = new Blob([JSON.stringify(errorInfo)], {
            type: 'application/json',
        });
        navigator.sendBeacon(url,blob)
    } else {
        console.log('senBeacon出问题了');
        //说是可以用fetch
    }

    // fetch(url,{
    //     method:'POST',
    //     headers: { 'Content-Type': 'application/json' },
    //     body:JSON.stringify(errorInfo)
    // })
}
```

然后要注意的是，这个Navigator.sendBeacon发送请求的时候是带着cookies的，后端设置跨域的时候需要动态设置跨域，不允许使用 * 通配符
