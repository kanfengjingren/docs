watch和computed ​

computed

ts
const route = useRoute();
const sessionId = computed(() => Number(route.params.id));

watch

ts
const route = useRoute();
let sessionId: number

watch(route, () => {
    sessionId = Number(route.params.id)
    console.log(route.params.id);

}, {
    immediate: true
})

这两个看上去的效果是一样的，都实现了切换路由的时候session的更新，但是其实是不一样的,或者说是监听的属性不一样，下面的要是换成ref就是一样的了

watch和watcheffect的区别 ​

  1. watch控制更细致，可以指定监听一个对象，而watcheffect是监听你里面所有的对象
  2. watcheffect会自动调用，而watch只在监听的值发生改变的时候调用 但是watch也可以通过immediate: true 启用默认执行一次
  3. watcheffect是深度监听的，就是ref对象的子属性改变了也会呗监听到，而watch要通过deep来开启
  4. watch可以通过oldvalue来获取旧的值而watcheffect不行

watch ​

有时候根据一些响应式数据变化，我们需要做出一些和这个响应式数据没有直接关系的其他操作，比如重新赋值，可以使用watch侦听响应式数据的变化 watch接收一个侦听对象和一个回调函数

  • 侦听对象可以包括：ref响应式数据、响应式对象、getter函数。
js
const x = ref(0)
const y = ref(0)

// 单个 ref
watch(x, (newX) => {
  console.log(`x is ${newX}`)
})

// getter 函数
watch(
  () => x.value + y.value,
  (sum) => {
    console.log(`sum of x + y is: ${sum}`)
  }
)

// 多个来源组成的数组
watch([x, () => y.value], ([newX, newY]) => {
  console.log(`x is ${newX} and y is ${newY}`)
})

但是watch不能直接侦听响应式对象的属性值,因为watch期待获得的是一个对象而不是值，所以你可以使用getter函数去返回这个值

ts
const obj = reactive({ count: 0 })

// 错误，因为 watch() 得到的参数是一个 number,而需要传入的是一个响应式对象
watch(obj.count, (count) => {
  console.log(`Count is: ${count}`)
})
ts
watch(
  () => obj.count,
  (count) => {
    console.log(`Count is: ${count}`)
  }
)
  • 深层调用 一般来说watch会默认开启深层侦听，就是侦听你这个响应式对象里面所有的变化，比如对象数组里面某一个对象的变化，但是你使用getter的时候，watch只会监听表层的变化：即你有没有对这个对象做出改变
ts
//这里有个pinia里面的对象 main.obj = {value:"2"}

watch(()=>main.obj,(newVal,oldVal)=>{
  console.log("改变了");
  console.log("newVal:"+newVal);
  console.log("oldVal:"+oldVal);
  count.value++
  
},{
  // deep:true
})
  1. 此时没有开启deep深层监听，如果你对这个pinia对象进行了一次重新赋值：比如说main.obj = {value:"2"},他是会触发的。如果你改成 main.obj.value = "3",这个时候是不会触发监听的。因为getter返回的是对象的应用
  2. 如果我们开启了深层监听，这个时候你就可以进行main.obj.value = "3"修改啦
  • 回调函数中有两个参数：(newVal,oldVal) 新值和旧值，不过也要注意如果是对象的话，那打印的都是一样的

watch里面额外的设置 ​

除了deep之外，还有一些参数可以使用：

  • immediate: true 立即调用一次
  • once: true 只执行一次
  • flush: 'post' 在组件更新后执行
  • flush: 'sync' 在组件更新之前执行

watch踩到的坑 ​

在我的这个代码里面有个坑,在渲染一个页面数据的时候，我需要通过路由去请求对应的数据赋值给main.chatHistory，还需要监听一下这个main.chatHistory去做一个页面的滚动。main.chatHistory是一个对象数组，存在pinia里面的

ts
watch(sessionId, async () => {
    main.chatHistory = await getChatHistory(sessionId.value)  
    //注意看这里重新赋值 main.chatHistory丢失响应式了
    console.log(isReactive(main.chatHistory));
    
    main.activeSession = sessionId.value;
}, {
    immediate: true
})//开头就调用一次

watch(()=>main.chatHistory, () => {
    console.log("检测到输入");

    scrollToBottom(scrollBottom)

    console.log(main.chatHistory);
    
}, {
    //deep: true, //deep是监听整个内容，没错，所以一切都说的通了，威慑呢么这里使用getter也会导致push的时候触发，就是因为使用了deep
    flush: "post"
})

由于上面的覆盖，导致这个丢失了响应式，当时不熟悉概念，什么deep、post一股脑全用上去了，结果发现使用getter竟然真的能监听到所有变化，不管是一开始的数组赋值变化，还是后面的push变化。现在一看就很清楚了

  • 数组赋值变化是对的：因为从响应式到被赋值了一个非相应式数组，本来到这里就该停了
  • 但是开了deep，所以监听到了后面这个数组的push操作，果然把deep怪了，就只会变化一次了

实际上应该怎么操作呢?

  1. 更改数组的赋值方式
ts
watch(sessionId, async () => {
  const newData = await getChatHistory(sessionId.value)
  main.chatHistory.length = 0
  main.chatHistory.push(...newData)
  main.activeSession = sessionId.value
}, { immediate: true })
  1. 可以监听main.chathistory身上的length啊，不过这样也需要getter函数返回，因为watch不能直接监听对象身上的属性
